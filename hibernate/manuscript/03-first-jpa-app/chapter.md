## Your First Entity: The Student Class

An **entity** is a Java class that maps to a database table. Each instance of the entity represents a row in that table. Each field in the class maps to a column.

Let us start with a simple example — a `Student` entity.

### What We Want to Achieve

```
Java Object                          Database Table: students
+--------------------+               +------+----------+-------------------+------+
| Student            |               | id   | name     | email             | age  |
|--------------------|     Maps      |------|----------|-------------------|------|
| id: 1              |  =========>   | 1    | Alice    | alice@example.com | 20   |
| name: "Alice"      |               | 2    | Bob      | bob@example.com   | 22   |
| email: "alice@..." |               | 3    | Charlie  | charlie@ex.com    | 21   |
| age: 20            |               +------+----------+-------------------+------+
+--------------------+
```

### Creating the Entity Class

Create the file `src/main/java/com/example/hibernatedemo/entity/Student.java`:

```java
package com.example.hibernatedemo.entity;

import jakarta.persistence.Entity;
import jakarta.persistence.GeneratedValue;
import jakarta.persistence.GenerationType;
import jakarta.persistence.Id;
import jakarta.persistence.Table;
import jakarta.persistence.Column;

@Entity                                    // JPA: Marks this class as a database entity
@Table(name = "students")                  // JPA: Maps to the "students" table
public class Student {

    @Id                                    // JPA: This field is the primary key
    @GeneratedValue(strategy =             // JPA: Database generates the ID
        GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false,              // JPA: Maps to "name" column, NOT NULL
            length = 100)
    private String name;

    @Column(nullable = false,              // JPA: Maps to "email" column, UNIQUE
            unique = true)
    private String email;

    private int age;                       // No @Column = uses defaults

    // Default constructor (REQUIRED by JPA)
    public Student() {
    }

    // Convenience constructor
    public Student(String name, String email, int age) {
        this.name = name;
        this.email = email;
        this.age = age;
    }

    // Getters and Setters
    public Long getId() {
        return id;
    }

    public void setId(Long id) {
        this.id = id;
    }

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public String getEmail() {
        return email;
    }

    public void setEmail(String email) {
        this.email = email;
    }

    public int getAge() {
        return age;
    }

    public void setAge(int age) {
        this.age = age;
    }

    @Override
    public String toString() {
        return "Student{id=" + id + ", name='" + name + "', email='" + email + "', age=" + age + "}";
    }
}
```

### Annotation Breakdown

Let us examine each annotation and what it does:

```
Annotation                    What It Does                     SQL Equivalent
---------------------------------------------------------------------------------
@Entity                       Marks class as a JPA entity      (tells Hibernate
                              (required for persistence)       to manage this class)

@Table(name = "students")     Maps class to table name         CREATE TABLE students

@Id                           Marks the primary key field      PRIMARY KEY

@GeneratedValue(strategy =    Auto-generate ID values          IDENTITY / AUTO_INCREMENT
  GenerationType.IDENTITY)

@Column(nullable = false,     Column constraints               name VARCHAR(100) NOT NULL
  length = 100)

@Column(unique = true)        Unique constraint                email VARCHAR(255) UNIQUE
```

### What Hibernate Generates

When you start the application with `ddl-auto=create-drop`, Hibernate reads the annotations and generates this SQL:

```sql
Hibernate:
    create table students (
        age integer not null,
        id bigint generated by default as identity,
        email varchar(255) not null unique,
        name varchar(100) not null,
        primary key (id)
    )
```

Notice how the annotations translated directly into SQL constraints. This is the ORM at work — you described the table structure in Java, and Hibernate created the SQL.

### The Three Requirements for a JPA Entity

Every JPA entity must satisfy three requirements:

```
+---------------------------------------------------------------+
|                  JPA Entity Requirements                       |
|                                                                |
|  1. @Entity annotation on the class                            |
|     (tells JPA this is a managed entity)                       |
|                                                                |
|  2. @Id annotation on a field                                  |
|     (every entity must have a primary key)                     |
|                                                                |
|  3. A public or protected no-argument constructor              |
|     (JPA creates instances using reflection)                   |
|                                                                |
|  Note: The class must NOT be final, and                        |
|        the fields must NOT be final                             |
|        (Hibernate creates proxy subclasses)                    |
+---------------------------------------------------------------+
```

If you forget any of these, you will get an error at startup. The most common mistake is forgetting the no-argument constructor (the default constructor disappears when you add any other constructor).

## Understanding the EntityManager

Before we use Spring Data JPA's convenient `JpaRepository`, let us understand what happens underneath. The **EntityManager** is the core JPA interface for all database operations. Spring Data JPA uses it internally, so understanding it will help you debug issues and write custom queries later.

### What the EntityManager Does

```
Your Code              EntityManager              Database
---------              -------------              --------

persist(student)  -->  Tracks the object     -->  INSERT INTO students ...
                       in the persistence
                       context (cache)

find(Student, 1)  -->  Checks cache first    -->  SELECT * FROM students
                       Returns from cache          WHERE id = 1
                       if found

merge(student)    -->  Copies state from     -->  UPDATE students SET ...
                       detached object to          WHERE id = 1
                       managed object

remove(student)   -->  Marks for deletion    -->  DELETE FROM students
                                                   WHERE id = 1
```

### The Persistence Context

The EntityManager maintains a **persistence context** — a first-level cache that tracks all entity objects during a transaction. This is one of JPA's most important concepts:

```
+-------------------------------------------------------------------+
|                    Persistence Context                              |
|                    (First-Level Cache)                              |
|                                                                    |
|  Transaction begins                                                |
|  |                                                                 |
|  |  persist(student1)  --> student1 is now "managed"               |
|  |                         (tracked by EntityManager)              |
|  |                                                                 |
|  |  find(Student, 1)   --> returns student1 from cache             |
|  |                         (NO SQL executed!)                      |
|  |                                                                 |
|  |  student1.setName("New Name")                                   |
|  |                     --> EntityManager detects the change         |
|  |                         automatically ("dirty checking")        |
|  |                                                                 |
|  Transaction commits                                               |
|  |                                                                 |
|  |  EntityManager flushes all changes to the database:             |
|  |  --> INSERT INTO students ... (for persist)                     |
|  |  --> UPDATE students SET name = 'New Name' ... (for setName)    |
|  |                                                                 |
|  Persistence context is cleared                                    |
+-------------------------------------------------------------------+
```

Key behaviors to remember:

1. **`persist()` does not immediately execute SQL**. It adds the entity to the persistence context. SQL runs at flush time (usually when the transaction commits).

2. **`find()` checks the cache first**. If you call `find(Student.class, 1L)` twice in the same transaction, the second call returns the cached object without hitting the database.

3. **Changes are detected automatically**. If you modify a managed entity (`student.setName("...")`), Hibernate detects the change and generates an UPDATE — you do not need to call any save method.

### Using EntityManager Directly

While Spring Data JPA is the recommended approach, understanding EntityManager helps you grasp what happens underneath:

```java
package com.example.hibernatedemo.repository;

import com.example.hibernatedemo.entity.Student;
import jakarta.persistence.EntityManager;
import jakarta.persistence.PersistenceContext;
import org.springframework.stereotype.Repository;
import org.springframework.transaction.annotation.Transactional;

@Repository
@Transactional
public class StudentEntityManagerRepository {

    @PersistenceContext                      // JPA: Injects the EntityManager
    private EntityManager entityManager;

    // CREATE
    public Student save(Student student) {
        entityManager.persist(student);      // Adds to persistence context
        return student;                      // ID is set after flush
    }

    // READ
    public Student findById(Long id) {
        return entityManager.find(Student.class, id);  // Returns null if not found
    }

    // UPDATE (managed entities are updated automatically)
    public Student update(Student student) {
        return entityManager.merge(student);  // Merges detached entity
    }

    // DELETE
    public void delete(Long id) {
        Student student = entityManager.find(Student.class, id);
        if (student != null) {
            entityManager.remove(student);    // Marks for deletion
        }
    }
}
```

This works, but it is a lot of boilerplate for basic CRUD. That is exactly what Spring Data JPA eliminates.

## Creating a Spring Data JPA Repository

Spring Data JPA lets you define a repository as a simple interface. Spring creates the implementation automatically at runtime.

### The Repository Interface

Create `src/main/java/com/example/hibernatedemo/repository/StudentRepository.java`:

```java
package com.example.hibernatedemo.repository;

import com.example.hibernatedemo.entity.Student;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;

import java.util.Optional;

@Repository
public interface StudentRepository extends JpaRepository<Student, Long> {

    // Spring Data JPA generates the implementation automatically!
    // You get all of these methods for free:
    //
    // save(Student)          - Create or update
    // findById(Long)         - Find by primary key
    // findAll()              - Find all students
    // deleteById(Long)       - Delete by primary key
    // count()                - Count all students
    // existsById(Long)       - Check if exists
    // ... and more
}
```

That is it. One interface, zero implementation code. Let us understand what is happening:

### How Spring Data JPA Works

```
What You Write:
+-------------------------------------------+
|  public interface StudentRepository       |
|      extends JpaRepository<Student, Long> |
|  { }                                      |
+-------------------------------------------+
        |
        | Spring Boot scans for JpaRepository interfaces
        | at startup and generates implementations
        v
What Spring Creates (at runtime):
+-------------------------------------------+
|  public class StudentRepositoryImpl       |
|      implements StudentRepository {       |
|                                           |
|    @PersistenceContext                     |
|    EntityManager em;                      |
|                                           |
|    public Student save(Student s) {       |
|      if (s.getId() == null) {             |
|        em.persist(s);                     |
|      } else {                             |
|        em.merge(s);                       |
|      }                                    |
|      return s;                            |
|    }                                      |
|                                           |
|    public Optional<Student>               |
|        findById(Long id) {               |
|      Student s = em.find(                 |
|          Student.class, id);              |
|      return Optional.ofNullable(s);       |
|    }                                      |
|                                           |
|    // ... all other methods               |
|  }                                        |
+-------------------------------------------+
```

You never see or write this implementation. Spring generates it at startup using proxy patterns and the EntityManager.

### The JpaRepository Hierarchy

`JpaRepository` extends several parent interfaces, each adding more methods:

```
Repository<T, ID>                    (marker interface, no methods)
    |
    v
CrudRepository<T, ID>               (basic CRUD: save, findById, delete, count)
    |
    v
ListCrudRepository<T, ID>           (same but returns List instead of Iterable)
    |
    v
PagingAndSortingRepository<T, ID>   (adds pagination and sorting)
    |
    v
ListPagingAndSortingRepository      (pagination returning List)
    |
    v
JpaRepository<T, ID>                (adds flush, saveAndFlush, batch operations)
```

By extending `JpaRepository<Student, Long>`, you get all methods from every interface in the chain. The two type parameters are:
- `Student` — the entity class
- `Long` — the type of the primary key (`@Id` field)

## Performing CRUD Operations

Now let us put it all together. We will create a `CommandLineRunner` that demonstrates all four CRUD operations and shows the SQL Hibernate generates.

### Loading Initial Data

Create `src/main/java/com/example/hibernatedemo/DataLoader.java`:

```java
package com.example.hibernatedemo;

import com.example.hibernatedemo.entity.Student;
import com.example.hibernatedemo.repository.StudentRepository;
import org.springframework.boot.CommandLineRunner;
import org.springframework.stereotype.Component;

@Component
public class DataLoader implements CommandLineRunner {

    private final StudentRepository studentRepository;

    public DataLoader(StudentRepository studentRepository) {
        this.studentRepository = studentRepository;
    }

    @Override
    public void run(String... args) {
        System.out.println("=== Starting Data Loader ===\n");

        // ---- CREATE ----
        System.out.println("--- CREATE: Saving students ---");

        Student alice = new Student("Alice Johnson", "alice@example.com", 20);
        Student bob = new Student("Bob Smith", "bob@example.com", 22);
        Student charlie = new Student("Charlie Brown", "charlie@example.com", 21);

        studentRepository.save(alice);
        studentRepository.save(bob);
        studentRepository.save(charlie);

        System.out.println("Saved: " + alice);   // ID is now set!
        System.out.println("Saved: " + bob);
        System.out.println("Saved: " + charlie);

        // ---- READ ----
        System.out.println("\n--- READ: Finding students ---");

        // Find by ID
        studentRepository.findById(1L).ifPresent(student ->
            System.out.println("Found by ID 1: " + student)
        );

        // Find all
        System.out.println("\nAll students:");
        studentRepository.findAll().forEach(student ->
            System.out.println("  " + student)
        );

        // Count
        long count = studentRepository.count();
        System.out.println("Total students: " + count);

        // ---- UPDATE ----
        System.out.println("\n--- UPDATE: Modifying a student ---");

        studentRepository.findById(1L).ifPresent(student -> {
            System.out.println("Before update: " + student);
            student.setAge(21);
            student.setEmail("alice.johnson@example.com");
            studentRepository.save(student);
            System.out.println("After update:  " + student);
        });

        // ---- DELETE ----
        System.out.println("\n--- DELETE: Removing a student ---");

        System.out.println("Exists before delete (ID 3): " +
            studentRepository.existsById(3L));
        studentRepository.deleteById(3L);
        System.out.println("Exists after delete (ID 3):  " +
            studentRepository.existsById(3L));

        // Final state
        System.out.println("\n--- Final state ---");
        studentRepository.findAll().forEach(student ->
            System.out.println("  " + student)
        );
        System.out.println("Total students: " + studentRepository.count());

        System.out.println("\n=== Data Loader Complete ===");
    }
}
```

## Understanding save() in Detail

The `save()` method in Spring Data JPA does something clever — it decides whether to INSERT or UPDATE based on the entity's ID:

```
repository.save(entity)
    |
    +-- Is entity.getId() == null?
    |       |
    |       +-- YES: This is a NEW entity
    |       |       --> entityManager.persist(entity)
    |       |       --> SQL: INSERT INTO students ...
    |       |       --> Generated ID is set on the entity
    |       |
    |       +-- NO: This is an EXISTING entity
    |               --> entityManager.merge(entity)
    |               --> SQL: SELECT ... (load current state)
    |               --> SQL: UPDATE students SET ... WHERE id = ?
    |
    +-- Returns the saved/updated entity
```

This means:

```java
// CREATE: id is null, so save() does INSERT
Student student = new Student("Alice", "alice@example.com", 20);
studentRepository.save(student);  // INSERT
// student.getId() is now 1 (set by Hibernate)

// UPDATE: id is 1 (not null), so save() does UPDATE
student.setAge(21);
studentRepository.save(student);  // UPDATE
```

### persist() vs merge()

This distinction is important for understanding JPA behavior:

```
+--------------------------------------------------------------------+
|  persist()                      |  merge()                          |
|  (for NEW entities)             |  (for EXISTING/DETACHED entities) |
|                                 |                                   |
|  - Entity must NOT have an ID   |  - Entity must have an ID         |
|  - Makes the entity "managed"   |  - Returns a NEW managed copy     |
|  - Throws exception if entity   |  - The original object remains    |
|    already has an ID (some       |    detached                       |
|    providers)                   |  - If entity does not exist in    |
|  - More efficient for inserts   |    DB, creates a new one          |
+--------------------------------------------------------------------+
```

Spring Data JPA's `save()` method handles this decision for you, which is why it is the preferred approach.

## Adding Custom Query Methods

Your `StudentRepository` currently only has the methods inherited from `JpaRepository`. You can add custom finder methods by following Spring Data JPA's naming conventions:

```java
package com.example.hibernatedemo.repository;

import com.example.hibernatedemo.entity.Student;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;

import java.util.List;
import java.util.Optional;

@Repository
public interface StudentRepository extends JpaRepository<Student, Long> {

    // Spring Data JPA generates the query from the method name!

    // Find by exact email
    Optional<Student> findByEmail(String email);

    // Find all students with a specific name
    List<Student> findByName(String name);

    // Find students older than a given age
    List<Student> findByAgeGreaterThan(int age);

    // Find students by name containing a string (LIKE query)
    List<Student> findByNameContaining(String keyword);

    // Check if a student with this email exists
    boolean existsByEmail(String email);

    // Count students with a specific age
    long countByAge(int age);

    // Delete all students with a given name
    void deleteByName(String name);
}
```

### How Method Name Parsing Works

Spring Data JPA parses the method name and generates JPQL/SQL:

```
Method Name                          Generated SQL
----------------------------------------------------------------------
findByEmail(email)                   SELECT ... WHERE email = ?

findByName(name)                     SELECT ... WHERE name = ?

findByAgeGreaterThan(age)            SELECT ... WHERE age > ?

findByNameContaining(keyword)        SELECT ... WHERE name LIKE '%keyword%'

existsByEmail(email)                 SELECT count(*) > 0
                                     WHERE email = ?

countByAge(age)                      SELECT count(*)
                                     WHERE age = ?

deleteByName(name)                   DELETE FROM students
                                     WHERE name = ?
```

### Method Name Keywords

```
Keyword                Example                          SQL Clause
----------------------------------------------------------------------
And                    findByNameAndAge                 WHERE name=? AND age=?
Or                     findByNameOrEmail                WHERE name=? OR email=?
Is, Equals             findByNameIs                     WHERE name=?
Between                findByAgeBetween                 WHERE age BETWEEN ? AND ?
LessThan               findByAgeLessThan                WHERE age < ?
GreaterThan            findByAgeGreaterThan             WHERE age > ?
Like                   findByNameLike                   WHERE name LIKE ?
Containing             findByNameContaining             WHERE name LIKE '%?%'
StartingWith           findByNameStartingWith           WHERE name LIKE '?%'
EndingWith             findByNameEndingWith             WHERE name LIKE '%?'
OrderBy                findByAgeOrderByNameAsc          ... ORDER BY name ASC
Not                    findByNameNot                    WHERE name <> ?
In                     findByAgeIn(Collection)          WHERE age IN (?, ?, ?)
True/False             findByActiveTrue                 WHERE active = true
IsNull                 findByEmailIsNull                WHERE email IS NULL
IsNotNull              findByEmailIsNotNull             WHERE email IS NOT NULL
```

## EntityManager vs JpaRepository: When to Use Which

You now know two ways to interact with the database. Here is when to use each:

```
+-----------------------------------------------------------------------+
|  Approach             | Use When                                       |
|-----------------------|------------------------------------------------|
|  JpaRepository        | - Standard CRUD operations                     |
|  (Spring Data JPA)    | - Derived query methods (findByName, etc.)     |
|                       | - Pagination and sorting                       |
|                       | - 95% of your data access needs                |
|                       |                                                |
|  EntityManager        | - Complex custom queries                       |
|  (JPA core)           | - Batch operations                             |
|                       | - Direct control over persistence context       |
|                       | - Stored procedure calls                       |
|                       | - When you need flush/clear/detach             |
+-----------------------------------------------------------------------+
```

In this book, we will primarily use `JpaRepository` because it handles the vast majority of use cases with minimal code. We will bring in `EntityManager` when we need fine-grained control (Chapter 17: Entity Lifecycle and Persistence Context).

        Student s2 = repository.save(new Student("Bob Smith", "bob@university.com", 22));

            s1.getId() + ", " + s2.getId() + ", " + s3.getId() + ", " +
            s4.getId() + ", " + s5.getId());

        System.out.println("\nAll students:");

        System.out.println("\nFind by ID 2:");

        System.out.println("\nFind by email 'charlie@university.com':");
        repository.findByEmail("charlie@university.com")
            .ifPresent(s -> System.out.println("  Found: " + s));

        System.out.println("\nStudents with 'a' in name:");
        repository.findByNameContaining("a")
            .forEach(s -> System.out.println("  " + s));

        System.out.println("\nStudents older than 21:");
        repository.findByAgeGreaterThan(21)
            .forEach(s -> System.out.println("  " + s));

        System.out.println("\nStudents aged 20-21:");
        repository.findByAgeBetween(20, 21)
            .forEach(s -> System.out.println("  " + s));

        System.out.println("\nTotal count: " + repository.count());
        System.out.println("Count aged 20: " + repository.countByAge(20));
        System.out.println("Email exists (alice@university.com): " +

        System.out.println("Before delete - count: " + repository.count());
        repository.deleteById(5L);

        // === FINAL STATE ===
        System.out.println("\n========== FINAL STATE ==========");
        repository.findAll().forEach(s -> System.out.println("  " + s));

## Common Mistakes

1. **Forgetting the no-argument constructor**: JPA creates entity instances using reflection, which requires a no-arg constructor. If you define any constructor, Java no longer provides the default one, so you must add it explicitly:

```java
// WRONG: Only has the parameterized constructor
@Entity
public class Student {
    public Student(String name, String email) { ... }
    // Crash! JPA cannot create instances
}

// CORRECT: Also has a no-arg constructor
@Entity
public class Student {
    public Student() {}  // Required by JPA
    public Student(String name, String email) { ... }
}
```

2. **Making the entity class or fields `final`**: Hibernate creates proxy subclasses for lazy loading. If the class is `final`, it cannot be subclassed:

```java
// WRONG
@Entity
public final class Student { ... }

// WRONG
@Entity
public class Student {
    private final String name;  // Cannot be set by reflection
}
```

3. **Not using `Optional` with `findById()`**: `findById()` returns `Optional<Student>`, not `Student`. Always handle the case where the entity might not exist:

```java
// WRONG: Can throw NoSuchElementException
Student student = repository.findById(999L).get();

// CORRECT: Handle the missing case
repository.findById(999L).ifPresentOrElse(
    student -> System.out.println("Found: " + student),
    () -> System.out.println("Student not found")
);

// ALSO CORRECT: With orElseThrow
Student student = repository.findById(999L)
    .orElseThrow(() -> new RuntimeException("Student not found"));
```

4. **Expecting save() to always do INSERT**: `save()` does INSERT when the ID is null and UPDATE (via merge) when the ID is not null. If you create a `new Student()` and manually set its ID, `save()` will try to UPDATE, not INSERT.

5. **Not looking at the generated SQL**: Always have `spring.jpa.show-sql=true` during development. If a simple operation is generating 10 SQL queries, something is wrong (probably an N+1 problem, covered in Chapter 18).

## Best Practices

1. **Always use `JpaRepository` as your starting point**: It provides everything you need for basic operations. Only drop down to `EntityManager` when you need fine-grained control.

2. **Use Optional returns for single-entity queries**: `findById()` returns `Optional<Student>`, which forces callers to handle the null case. Define your custom finder methods to return `Optional` for single results:

```java
Optional<Student> findByEmail(String email);  // Good
Student findByEmail(String email);            // Returns null if not found (risky)
```

3. **Use constructor injection for repositories**: Always inject repositories through the constructor, not field injection:

```java
// GOOD: Constructor injection
private final StudentRepository repository;
public DataLoader(StudentRepository repository) {
    this.repository = repository;
}

// AVOID: Field injection
@Autowired
private StudentRepository repository;
```

4. **Keep entities simple**: Entity classes should only contain field mappings, getters, setters, and `toString()`. Business logic belongs in service classes.

5. **Name your tables explicitly**: Use `@Table(name = "students")` rather than relying on the default naming strategy. This makes your code self-documenting and avoids surprises when entity names change.

## Summary

In this chapter, you built your first complete JPA application:

- **Entity classes** use `@Entity`, `@Id`, `@GeneratedValue`, `@Column`, and `@Table` annotations to map Java objects to database tables. These are all JPA standard annotations from `jakarta.persistence`.

- **EntityManager** is the core JPA interface. It provides `persist()`, `find()`, `merge()`, and `remove()` for direct database operations. Understanding it helps you debug JPA issues.

- **The persistence context** is a first-level cache managed by EntityManager. It tracks entity state changes and flushes them to the database at transaction commit time.

- **JpaRepository** is a Spring Data JPA interface that auto-implements all basic CRUD operations. You define an interface; Spring generates the implementation at startup.

- **Derived query methods** let you define custom queries by naming your method according to conventions (e.g., `findByEmail`, `findByAgeGreaterThan`). Spring parses the method name and generates the query.

- **save()** is smart — it calls `persist()` for new entities (null ID) and `merge()` for existing entities (non-null ID).

- **SQL logging** (`spring.jpa.show-sql=true`) shows you exactly what Hibernate does for each operation. Always enable it during development.

## Interview Questions

**Q1: What are the minimum requirements for a JPA entity class?**

A JPA entity must have: (1) the `@Entity` annotation on the class, (2) an `@Id` annotation on a field to define the primary key, and (3) a public or protected no-argument constructor. Additionally, the class and its persistent fields must not be `final`.

**Q2: What is the difference between `persist()` and `merge()` in JPA?**

`persist()` is for new entities that are not yet in the database. It makes the entity managed and schedules an INSERT. `merge()` is for detached entities (previously managed but no longer tracked). It copies the state of the detached entity onto a new managed instance and schedules an UPDATE. `persist()` modifies the original object; `merge()` returns a new managed copy.

**Q3: How does Spring Data JPA's `save()` method decide whether to INSERT or UPDATE?**

It checks if the entity is "new" by examining the `@Id` field. If the ID is `null` (or zero for primitive types), the entity is considered new, and `persist()` is called (INSERT). If the ID has a value, `merge()` is called (UPDATE). This behavior can be customized by implementing the `Persistable` interface.

**Q4: What is the persistence context in JPA?**

The persistence context is a first-level cache managed by the EntityManager within a transaction. It tracks all managed entity instances. If you load the same entity twice in a transaction, the second call returns the cached instance. It also performs "dirty checking" — detecting changes to managed entities and automatically generating UPDATE statements at flush time.

**Q5: How do derived query methods work in Spring Data JPA?**

Spring Data JPA parses the method name to build a query. For example, `findByNameContaining(String keyword)` is parsed as: `find` (query type) + `By` (predicate start) + `Name` (field name) + `Containing` (LIKE with wildcards). It generates: `SELECT ... FROM students WHERE name LIKE '%keyword%'`. The method must follow specific naming conventions documented by Spring Data.

**Q6: Why does `deleteById()` execute a SELECT before the DELETE?**

JPA requires an entity to be in the managed state before it can be removed. `deleteById()` first loads the entity using `findById()` (SELECT), makes it managed, then calls `remove()` (DELETE). This ensures lifecycle callbacks (`@PreRemove`, `@PostRemove`) are triggered. If you want to delete without loading the entity first, you can use a custom `@Query` with a DELETE statement.

**Q7: What is the difference between `JpaRepository` and `CrudRepository`?**

`CrudRepository` provides basic CRUD operations: `save`, `findById`, `findAll`, `delete`, `count`, `existsById`. `JpaRepository` extends `CrudRepository` and adds JPA-specific methods: `flush()`, `saveAndFlush()`, `saveAllAndFlush()`, `deleteInBatch()`, `deleteAllInBatch()`, `getById()`. It also extends `PagingAndSortingRepository` for pagination. For most applications, `JpaRepository` is the right choice.
