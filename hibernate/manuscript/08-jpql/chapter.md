## What Is JPQL?

**JPQL (Java Persistence Query Language)** is a query language defined by the JPA specification. It looks similar to SQL, but with one fundamental difference: **JPQL queries entities and their fields, not tables and columns**.

```
+------------------------------------------------------------------+
|  SQL vs JPQL                                                      |
|                                                                   |
|  SQL:   SELECT * FROM employees WHERE department = 'ENGINEERING'  |
|         ^queries tables and columns                               |
|                                                                   |
|  JPQL:  SELECT e FROM Employee e WHERE e.department = 'ENGINEERING'|
|         ^queries entities and fields                              |
|                                                                   |
|  Key differences:                                                 |
|  - SQL uses table names          JPQL uses entity class names     |
|  - SQL uses column names         JPQL uses field names            |
|  - SQL is database-specific      JPQL is database-independent     |
|  - SQL returns rows              JPQL returns entity objects      |
+------------------------------------------------------------------+
```

### Why JPQL Instead of SQL?

```
JPQL Advantages:
+------------------------------------------------------------------+
|  1. Database-independent                                          |
|     Same query works on H2, PostgreSQL, MySQL, Oracle             |
|     Hibernate translates JPQL to the correct SQL dialect          |
|                                                                   |
|  2. Object-oriented                                               |
|     Queries reference entity fields, not columns                  |
|     Navigate relationships with dot notation: e.department.name   |
|                                                                   |
|  3. Type-safe with compile-time checking                          |
|     Field names are validated at startup                          |
|     Typos are caught early, not at runtime                        |
|                                                                   |
|  4. Automatic mapping                                             |
|     Results are automatically mapped to entity objects             |
|     No manual ResultSet parsing                                   |
+------------------------------------------------------------------+
```

## JPQL Syntax Overview

### The Basic Structure

```
SELECT [what to return]
FROM [EntityName] [alias]
WHERE [conditions]
GROUP BY [fields]
HAVING [group conditions]
ORDER BY [fields] [ASC|DESC]
```

### JPQL vs SQL Side-by-Side

```
JPQL                                     SQL Generated by Hibernate
--------------------------------------  --------------------------------------
SELECT e                                 SELECT e.id, e.name, e.email,
FROM Employee e                                 e.department, e.salary,
                                                e.active
                                         FROM employees e

SELECT e                                 SELECT e.id, e.name, e.email,
FROM Employee e                                 e.department, e.salary,
WHERE e.department = :dept                      e.active
                                         FROM employees e
                                         WHERE e.department = ?

SELECT e.name, e.salary                  SELECT e.name, e.salary
FROM Employee e                          FROM employees e
WHERE e.salary > :min                    WHERE e.salary > ?
ORDER BY e.salary DESC                   ORDER BY e.salary DESC

SELECT COUNT(e)                          SELECT COUNT(e.id)
FROM Employee e                          FROM employees e
WHERE e.active = true                    WHERE e.active = true
```

Notice:
- `Employee` (JPQL) maps to `employees` (SQL table)
- `e.department` (JPQL field) maps to `e.department` (SQL column, via naming strategy)
- `SELECT e` returns the full entity; Hibernate expands it to all columns
- `:dept` (named parameter) becomes `?` in the actual SQL

## Setup: Entities for This Chapter

We will use two related entities to demonstrate joins:

```java
@Entity
@Table(name = "departments")
public class Department {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, unique = true, length = 50)
    private String name;

    @Column(length = 200)
    private String location;

    @OneToMany(mappedBy = "department")
    private List<Employee> employees = new ArrayList<>();

    public Department() {}
    public Department(String name, String location) {
        this.name = name;
        this.location = location;
    }

    // getters, setters...
}
```

```java
@Entity
@Table(name = "employees")
public class Employee {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 100)
    private String name;

    @Column(nullable = false, unique = true)
    private String email;

    @Column(precision = 10, scale = 2)
    private BigDecimal salary;

    private boolean active;

    private LocalDate hireDate;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "department_id")
    private Department department;

    public Employee() {}
    public Employee(String name, String email, BigDecimal salary,
                    boolean active, LocalDate hireDate) {
        this.name = name;
        this.email = email;
        this.salary = salary;
        this.active = active;
        this.hireDate = hireDate;
    }

    // getters, setters...
}
```

**Note**: We will cover `@ManyToOne` and `@OneToMany` in detail in Chapter 13. For now, just understand that an Employee belongs to a Department, and a Department has many Employees.

## Using @Query in Spring Data JPA

The `@Query` annotation lets you write JPQL (or native SQL) directly on repository methods:

```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // JPQL query using @Query
    @Query("SELECT e FROM Employee e WHERE e.salary > :minSalary")
    List<Employee> findHighEarners(@Param("minSalary") BigDecimal minSalary);

    // Equivalent derived method (for comparison):
    // List<Employee> findBySalaryGreaterThan(BigDecimal salary);
}
```

### Named Parameters vs Positional Parameters

```java
// Named parameters — recommended (clear and self-documenting)
@Query("SELECT e FROM Employee e WHERE e.name = :name AND e.active = :active")
List<Employee> findByNameAndStatus(@Param("name") String name,
                                   @Param("active") boolean active);

// Positional parameters — shorter but less readable
@Query("SELECT e FROM Employee e WHERE e.name = ?1 AND e.active = ?2")
List<Employee> findByNameAndStatus(String name, boolean active);
```

```
Named vs Positional Parameters:
+------------------------------------------------------------------+
|  Named (:name)                    Positional (?1)                 |
|  @Param("name") String name       String name  (position 1)      |
|                                                                   |
|  More readable                    Shorter                         |
|  Order does not matter            Order must match ?1, ?2, ?3     |
|  Self-documenting                 Fragile if params reordered     |
|  Recommended                      Acceptable for simple queries   |
+------------------------------------------------------------------+
```

**Best practice**: Always use named parameters. They make queries readable and safe against parameter reordering.

## SELECT Queries

### Selecting Full Entities

```java
// Select all employees
@Query("SELECT e FROM Employee e")
List<Employee> findAllEmployees();

// With a condition
@Query("SELECT e FROM Employee e WHERE e.active = true")
List<Employee> findActiveEmployees();

// The alias 'e' is required — JPQL needs it for field references
```

### Selecting Specific Fields

```java
// Select only name — returns List<String>
@Query("SELECT e.name FROM Employee e WHERE e.active = true")
List<String> findActiveEmployeeNames();

// Select two fields — returns List<Object[]>
@Query("SELECT e.name, e.salary FROM Employee e ORDER BY e.salary DESC")
List<Object[]> findNamesAndSalaries();
```

When selecting multiple fields, the return type is `Object[]`. You access fields by index:

```java
List<Object[]> results = repository.findNamesAndSalaries();
for (Object[] row : results) {
    String name = (String) row[0];
    BigDecimal salary = (BigDecimal) row[1];
    System.out.println(name + ": $" + salary);
}
```

This is awkward. A better approach is constructor expressions (covered later in this chapter).

### DISTINCT

```java
// Remove duplicates
@Query("SELECT DISTINCT e.department FROM Employee e")
List<Department> findDistinctDepartments();

@Query("SELECT DISTINCT e.name FROM Employee e WHERE e.active = true")
List<String> findDistinctActiveNames();
```

## WHERE Clause

### Comparison Operators

```java
// Equals
@Query("SELECT e FROM Employee e WHERE e.name = :name")
List<Employee> findByExactName(@Param("name") String name);

// Not equals
@Query("SELECT e FROM Employee e WHERE e.department <> :dept")
List<Employee> findNotInDepartment(@Param("dept") Department dept);

// Greater than, less than
@Query("SELECT e FROM Employee e WHERE e.salary > :min AND e.salary < :max")
List<Employee> findBySalaryRange(@Param("min") BigDecimal min,
                                 @Param("max") BigDecimal max);

// BETWEEN
@Query("SELECT e FROM Employee e WHERE e.salary BETWEEN :min AND :max")
List<Employee> findBySalaryBetween(@Param("min") BigDecimal min,
                                   @Param("max") BigDecimal max);
```

### String Operations

```java
// LIKE with wildcards
@Query("SELECT e FROM Employee e WHERE e.name LIKE :pattern")
List<Employee> findByNamePattern(@Param("pattern") String pattern);
// Usage: findByNamePattern("%Alice%")

// LIKE with CONCAT (add wildcards in the query)
@Query("SELECT e FROM Employee e WHERE e.name LIKE CONCAT('%', :keyword, '%')")
List<Employee> searchByName(@Param("keyword") String keyword);
// Usage: searchByName("Alice")  — no wildcards needed from caller

// UPPER / LOWER for case-insensitive
@Query("SELECT e FROM Employee e WHERE LOWER(e.name) LIKE LOWER(CONCAT('%', :keyword, '%'))")
List<Employee> searchByNameIgnoreCase(@Param("keyword") String keyword);
```

### NULL Checks

```java
@Query("SELECT e FROM Employee e WHERE e.department IS NULL")
List<Employee> findUnassignedEmployees();

@Query("SELECT e FROM Employee e WHERE e.department IS NOT NULL")
List<Employee> findAssignedEmployees();
```

### IN Clause

```java
@Query("SELECT e FROM Employee e WHERE e.department.name IN :deptNames")
List<Employee> findByDepartmentNames(@Param("deptNames") List<String> deptNames);
// Usage: findByDepartmentNames(List.of("Engineering", "Sales"))
// SQL: WHERE d.name IN (?, ?)
```

### Logical Operators

```java
// AND
@Query("SELECT e FROM Employee e WHERE e.active = true AND e.salary > :min")
List<Employee> findActiveHighEarners(@Param("min") BigDecimal min);

// OR
@Query("SELECT e FROM Employee e WHERE e.department.name = :dept1 OR e.department.name = :dept2")
List<Employee> findInEitherDepartment(@Param("dept1") String dept1,
                                      @Param("dept2") String dept2);

// NOT
@Query("SELECT e FROM Employee e WHERE NOT (e.active = true AND e.salary < :max)")
List<Employee> findNotActiveLowEarners(@Param("max") BigDecimal max);

// Complex conditions with parentheses
@Query("SELECT e FROM Employee e WHERE " +
       "(e.department.name = :dept AND e.salary > :min) OR " +
       "(e.active = false)")
List<Employee> findComplexCondition(@Param("dept") String dept,
                                    @Param("min") BigDecimal min);
```

## JOIN Queries

JPQL supports implicit joins (via dot navigation) and explicit joins.

### Implicit Join (Dot Navigation)

```java
// Navigate from Employee to Department using dot notation
@Query("SELECT e FROM Employee e WHERE e.department.name = :deptName")
List<Employee> findByDepartmentName(@Param("deptName") String deptName);
// Hibernate generates: SELECT ... FROM employees e
//                      JOIN departments d ON e.department_id = d.id
//                      WHERE d.name = ?
```

Hibernate automatically generates the JOIN when you navigate a relationship with dot notation. This is convenient but gives you less control.

### Explicit JOIN

```java
// Explicit inner join
@Query("SELECT e FROM Employee e JOIN e.department d WHERE d.name = :deptName")
List<Employee> findByDepartmentNameWithJoin(@Param("deptName") String deptName);

// Explicit inner join selecting from both entities
@Query("SELECT e.name, d.name FROM Employee e JOIN e.department d")
List<Object[]> findEmployeeAndDepartmentNames();
```

### LEFT JOIN

```java
// Left join — includes employees without a department
@Query("SELECT e FROM Employee e LEFT JOIN e.department d")
List<Employee> findAllWithOptionalDepartment();

// Left join with condition
@Query("SELECT e FROM Employee e LEFT JOIN e.department d " +
       "WHERE d.name = :dept OR d IS NULL")
List<Employee> findByDeptOrUnassigned(@Param("dept") String dept);
```

### JOIN FETCH (Performance Optimization)

`JOIN FETCH` loads the related entity in the same query, avoiding the N+1 problem:

```java
// Without JOIN FETCH — N+1 problem:
// Query 1: SELECT * FROM employees
// Query 2: SELECT * FROM departments WHERE id = 1  (for emp1)
// Query 3: SELECT * FROM departments WHERE id = 2  (for emp2)
// ... N additional queries for N employees!

// With JOIN FETCH — single query:
@Query("SELECT e FROM Employee e JOIN FETCH e.department")
List<Employee> findAllWithDepartment();
// Generates: SELECT e.*, d.* FROM employees e
//            JOIN departments d ON e.department_id = d.id
// ONE query loads everything!
```

```
N+1 Problem vs JOIN FETCH:
+------------------------------------------------------------------+
|  Without JOIN FETCH:                                              |
|  Query 1: SELECT * FROM employees           (returns 100 emps)   |
|  Query 2: SELECT * FROM departments WHERE id=1                   |
|  Query 3: SELECT * FROM departments WHERE id=2                   |
|  ...                                                              |
|  Query 101: SELECT * FROM departments WHERE id=100               |
|  Total: 101 queries!                                              |
|                                                                   |
|  With JOIN FETCH:                                                 |
|  Query 1: SELECT e.*, d.* FROM employees e                       |
|           JOIN departments d ON e.department_id = d.id            |
|  Total: 1 query!                                                  |
+------------------------------------------------------------------+
```

We will cover the N+1 problem in much greater depth in Chapter 18.

### LEFT JOIN FETCH

```java
// Left join fetch — includes employees without departments
@Query("SELECT e FROM Employee e LEFT JOIN FETCH e.department")
List<Employee> findAllWithOptionalDepartment();
```

## Aggregate Functions

JPQL supports the standard SQL aggregate functions:

```
Function     Purpose                Example
------------------------------------------------------
COUNT        Count rows             COUNT(e)
SUM          Sum values             SUM(e.salary)
AVG          Average value          AVG(e.salary)
MIN          Minimum value          MIN(e.salary)
MAX          Maximum value          MAX(e.salary)
```

### Basic Aggregations

```java
// Count all employees
@Query("SELECT COUNT(e) FROM Employee e")
long countAll();

// Count active employees
@Query("SELECT COUNT(e) FROM Employee e WHERE e.active = true")
long countActive();

// Sum of salaries
@Query("SELECT SUM(e.salary) FROM Employee e WHERE e.active = true")
BigDecimal totalActiveSalary();

// Average salary
@Query("SELECT AVG(e.salary) FROM Employee e")
Double averageSalary();

// Min and Max salary
@Query("SELECT MIN(e.salary) FROM Employee e")
BigDecimal minSalary();

@Query("SELECT MAX(e.salary) FROM Employee e")
BigDecimal maxSalary();

// Multiple aggregates in one query
@Query("SELECT COUNT(e), AVG(e.salary), MIN(e.salary), MAX(e.salary) " +
       "FROM Employee e WHERE e.active = true")
Object[] getSalaryStats();
// Access: Object[] stats = repository.getSalaryStats();
// long count = (long) stats[0];
// double avg = (double) stats[1];
// BigDecimal min = (BigDecimal) stats[2];
// BigDecimal max = (BigDecimal) stats[3];
```

## GROUP BY and HAVING

### GROUP BY

```java
// Count employees per department
@Query("SELECT d.name, COUNT(e) FROM Employee e " +
       "JOIN e.department d GROUP BY d.name")
List<Object[]> countByDepartment();

// Average salary per department
@Query("SELECT d.name, AVG(e.salary) FROM Employee e " +
       "JOIN e.department d GROUP BY d.name ORDER BY AVG(e.salary) DESC")
List<Object[]> avgSalaryByDepartment();

// Usage:
List<Object[]> results = repository.countByDepartment();
for (Object[] row : results) {
    String deptName = (String) row[0];
    Long count = (Long) row[1];
    System.out.println(deptName + ": " + count + " employees");
}
```

### HAVING (Filter Groups)

```java
// Departments with more than 5 employees
@Query("SELECT d.name, COUNT(e) FROM Employee e " +
       "JOIN e.department d GROUP BY d.name HAVING COUNT(e) > :minCount")
List<Object[]> largeDepartments(@Param("minCount") long minCount);

// Departments with average salary above a threshold
@Query("SELECT d.name, AVG(e.salary) FROM Employee e " +
       "JOIN e.department d GROUP BY d.name HAVING AVG(e.salary) > :minAvg")
List<Object[]> highPayingDepartments(@Param("minAvg") double minAvg);
```

```
GROUP BY + HAVING Flow:
+------------------------------------------------------------------+
|  Data:                                                            |
|  Engineering: Alice(95K), Bob(105K), Charlie(88K)  = 3 employees  |
|  Marketing:   Diana(78K), Eve(82K)                 = 2 employees  |
|  Sales:       Frank(72K)                           = 1 employee   |
|                                                                   |
|  Query: SELECT d.name, COUNT(e), AVG(e.salary)                    |
|         FROM Employee e JOIN e.department d                        |
|         GROUP BY d.name                                            |
|         HAVING COUNT(e) >= 2                                       |
|                                                                   |
|  Result:                                                          |
|  +---------------+-------+----------+                             |
|  | Department    | Count | Avg Sal  |                             |
|  |---------------|-------|----------|                             |
|  | Engineering   | 3     | 96000.00 |                             |
|  | Marketing     | 2     | 80000.00 |                             |
|  +---------------+-------+----------+                             |
|  (Sales excluded — only 1 employee, fails HAVING COUNT >= 2)     |
+------------------------------------------------------------------+
```

## Constructor Expressions — Returning DTOs

Instead of returning `Object[]` for multi-field queries, you can use **constructor expressions** to return DTOs directly:

### Step 1: Create a DTO

```java
package com.example.hibernatedemo.dto;

public class EmployeeSummary {

    private String name;
    private String departmentName;
    private BigDecimal salary;

    // Constructor must match the JPQL SELECT fields exactly
    public EmployeeSummary(String name, String departmentName, BigDecimal salary) {
        this.name = name;
        this.departmentName = departmentName;
        this.salary = salary;
    }

    // Getters
    public String getName() { return name; }
    public String getDepartmentName() { return departmentName; }
    public BigDecimal getSalary() { return salary; }

    @Override
    public String toString() {
        return name + " (" + departmentName + ") - $" + salary;
    }
}
```

### Step 2: Use Constructor Expression in JPQL

```java
@Query("SELECT new com.example.hibernatedemo.dto.EmployeeSummary(" +
       "e.name, d.name, e.salary) " +
       "FROM Employee e JOIN e.department d " +
       "WHERE e.active = true ORDER BY e.salary DESC")
List<EmployeeSummary> findActiveEmployeeSummaries();
```

The `new com.example...EmployeeSummary(e.name, d.name, e.salary)` tells JPQL to create a DTO instance for each row, passing the selected fields to the constructor.

```
Constructor Expression vs Object[]:
+------------------------------------------------------------------+
|  Object[] (fragile):                                              |
|  Object[] row = results.get(0);                                   |
|  String name = (String) row[0];        // index-based, fragile    |
|  String dept = (String) row[1];        // wrong index = ClassCast |
|  BigDecimal sal = (BigDecimal) row[2]; // no compile-time safety  |
|                                                                   |
|  Constructor Expression (type-safe):                              |
|  EmployeeSummary summary = results.get(0);                        |
|  String name = summary.getName();      // type-safe               |
|  String dept = summary.getDepartmentName(); // self-documenting   |
|  BigDecimal sal = summary.getSalary(); // compile-time checked    |
+------------------------------------------------------------------+
```

### More DTO Examples

```java
// Department statistics DTO
public class DepartmentStats {
    private String departmentName;
    private long employeeCount;
    private double averageSalary;

    public DepartmentStats(String departmentName, long employeeCount,
                           double averageSalary) {
        this.departmentName = departmentName;
        this.employeeCount = employeeCount;
        this.averageSalary = averageSalary;
    }
    // getters...
}

// Query returning the DTO
@Query("SELECT new com.example.hibernatedemo.dto.DepartmentStats(" +
       "d.name, COUNT(e), AVG(e.salary)) " +
       "FROM Employee e JOIN e.department d " +
       "GROUP BY d.name ORDER BY COUNT(e) DESC")
List<DepartmentStats> getDepartmentStatistics();
```

## @NamedQuery — Reusable Entity-Level Queries

`@NamedQuery` defines queries on the entity class itself. They are validated at application startup (typos are caught early):

```java
@Entity
@Table(name = "employees")
@NamedQuery(
    name = "Employee.findByDepartmentName",
    query = "SELECT e FROM Employee e JOIN e.department d WHERE d.name = :deptName"
)
@NamedQuery(
    name = "Employee.findActiveHighEarners",
    query = "SELECT e FROM Employee e WHERE e.active = true AND e.salary > :minSalary ORDER BY e.salary DESC"
)
@NamedQuery(
    name = "Employee.countByDepartment",
    query = "SELECT d.name, COUNT(e) FROM Employee e JOIN e.department d GROUP BY d.name"
)
public class Employee {
    // ... entity fields
}
```

### Using @NamedQuery in Repositories

Spring Data JPA automatically resolves named queries if they follow the convention `EntityName.methodName`:

```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // Matches @NamedQuery(name = "Employee.findByDepartmentName")
    List<Employee> findByDepartmentName(@Param("deptName") String deptName);

    // Matches @NamedQuery(name = "Employee.findActiveHighEarners")
    List<Employee> findActiveHighEarners(@Param("minSalary") BigDecimal minSalary);
}
```

```
@NamedQuery vs @Query:
+------------------------------------------------------------------+
|  @NamedQuery                        @Query                        |
|  (on Entity class)                  (on Repository method)        |
|                                                                   |
|  Validated at startup               Validated at startup           |
|  Defined on the entity              Defined on the repository     |
|  Reusable across repositories       Tied to one method            |
|  Query separate from method         Query next to method          |
|  Convention: Entity.methodName       Explicit annotation           |
|                                                                   |
|  Use when: query is reused          Use when: query is specific   |
|  across multiple repositories       to one repository method      |
+------------------------------------------------------------------+
```

**Best practice**: Use `@Query` on repository methods for most cases — it keeps the query close to where it is used. Use `@NamedQuery` when the same query is needed in multiple places or you want centralized query management on the entity.

## UPDATE and DELETE in JPQL

JPQL supports bulk UPDATE and DELETE operations that execute as single SQL statements:

### Bulk UPDATE

```java
@Modifying
@Query("UPDATE Employee e SET e.salary = e.salary * :multiplier " +
       "WHERE e.department.name = :dept")
int adjustSalariesByDepartment(@Param("multiplier") BigDecimal multiplier,
                               @Param("dept") String dept);

@Modifying
@Query("UPDATE Employee e SET e.active = false WHERE e.hireDate < :cutoff")
int deactivateBefore(@Param("cutoff") LocalDate cutoff);

@Modifying(clearAutomatically = true)
@Query("UPDATE Employee e SET e.department = :newDept " +
       "WHERE e.department = :oldDept")
int transferDepartment(@Param("oldDept") Department oldDept,
                       @Param("newDept") Department newDept);
```

### Bulk DELETE

```java
@Modifying
@Query("DELETE FROM Employee e WHERE e.active = false")
int deleteInactiveEmployees();

@Modifying
@Query("DELETE FROM Employee e WHERE e.hireDate < :cutoff AND e.active = false")
int purgeOldInactiveEmployees(@Param("cutoff") LocalDate cutoff);
```

### Important Rules for @Modifying

```
+------------------------------------------------------------------+
|  @Modifying Rules                                                 |
|                                                                   |
|  1. Must return int or void                                       |
|     int = number of rows affected                                 |
|                                                                   |
|  2. Must be in a @Transactional context                           |
|     Either on the repository method or the calling service        |
|                                                                   |
|  3. Does NOT trigger lifecycle callbacks                          |
|     @PreUpdate, @PreRemove do NOT fire                            |
|                                                                   |
|  4. Does NOT update the persistence context                       |
|     Use clearAutomatically = true to avoid stale data             |
|                                                                   |
|  5. Uses JPQL syntax (entity/field names, not table/column)       |
|     UPDATE Employee e SET e.salary = ... (not UPDATE employees)   |
+------------------------------------------------------------------+
```

## JPQL Functions

JPQL provides built-in functions for string manipulation, math, and dates:

### String Functions

```java
// CONCAT
@Query("SELECT CONCAT(e.name, ' (', d.name, ')') FROM Employee e JOIN e.department d")
List<String> findFormattedNames();

// LENGTH
@Query("SELECT e FROM Employee e WHERE LENGTH(e.name) > :minLength")
List<Employee> findLongNames(@Param("minLength") int minLength);

// UPPER, LOWER
@Query("SELECT e FROM Employee e WHERE UPPER(e.name) = UPPER(:name)")
List<Employee> findByNameCaseInsensitive(@Param("name") String name);

// SUBSTRING
@Query("SELECT SUBSTRING(e.email, 1, LOCATE('@', e.email) - 1) FROM Employee e")
List<String> findEmailUsernames();

// TRIM
@Query("SELECT e FROM Employee e WHERE TRIM(e.name) = :name")
List<Employee> findByTrimmedName(@Param("name") String name);

// LOCATE (find position of substring)
@Query("SELECT e FROM Employee e WHERE LOCATE('admin', e.email) > 0")
List<Employee> findAdminEmails();
```

### Math Functions

```java
// ABS
@Query("SELECT e FROM Employee e WHERE ABS(e.salary - :target) < :tolerance")
List<Employee> findBySalaryNear(@Param("target") BigDecimal target,
                                @Param("tolerance") BigDecimal tolerance);

// MOD
@Query("SELECT e FROM Employee e WHERE MOD(e.id, 2) = 0")
List<Employee> findWithEvenIds();

// SIZE (collection size)
@Query("SELECT d FROM Department d WHERE SIZE(d.employees) > :minSize")
List<Department> findLargeDepartments(@Param("minSize") int minSize);
```

### Date Functions

```java
// CURRENT_DATE, CURRENT_TIME, CURRENT_TIMESTAMP
@Query("SELECT e FROM Employee e WHERE e.hireDate < CURRENT_DATE")
List<Employee> findHiredBeforeToday();

// YEAR, MONTH, DAY (JPA 3.1+)
@Query("SELECT e FROM Employee e WHERE YEAR(e.hireDate) = :year")
List<Employee> findHiredInYear(@Param("year") int year);
```

## CASE Expressions

JPQL supports conditional expressions similar to SQL CASE:

```java
// Simple CASE
@Query("SELECT e.name, " +
       "CASE e.active WHEN true THEN 'Active' ELSE 'Inactive' END " +
       "FROM Employee e")
List<Object[]> findWithStatusLabel();

// Searched CASE (with conditions)
@Query("SELECT e.name, " +
       "CASE WHEN e.salary >= 100000 THEN 'Senior' " +
       "     WHEN e.salary >= 70000 THEN 'Mid' " +
       "     ELSE 'Junior' END " +
       "FROM Employee e ORDER BY e.salary DESC")
List<Object[]> findWithSalaryBand();

// CASE in WHERE clause
@Query("SELECT e FROM Employee e WHERE " +
       "CASE WHEN e.active = true THEN e.salary ELSE 0 END > :min")
List<Employee> findActiveAboveMinSalary(@Param("min") BigDecimal min);
```

### COALESCE and NULLIF

```java
// COALESCE — returns first non-null value
@Query("SELECT e.name, COALESCE(d.name, 'Unassigned') " +
       "FROM Employee e LEFT JOIN e.department d")
List<Object[]> findWithDepartmentOrDefault();

// NULLIF — returns null if values are equal
@Query("SELECT e.name, NULLIF(e.salary, 0) FROM Employee e")
List<Object[]> findSalariesNullIfZero();
```

## Subqueries

JPQL supports subqueries in WHERE and HAVING clauses:

```java
// Employees with salary above the average
@Query("SELECT e FROM Employee e WHERE e.salary > " +
       "(SELECT AVG(e2.salary) FROM Employee e2)")
List<Employee> findAboveAverageSalary();

// Employees in the largest department
@Query("SELECT e FROM Employee e WHERE e.department = " +
       "(SELECT e2.department FROM Employee e2 " +
       " GROUP BY e2.department ORDER BY COUNT(e2) DESC LIMIT 1)")
List<Employee> findInLargestDepartment();

// Departments that have at least one employee earning above a threshold
@Query("SELECT d FROM Department d WHERE d IN " +
       "(SELECT e.department FROM Employee e WHERE e.salary > :min)")
List<Department> findDepartmentsWithHighEarners(@Param("min") BigDecimal min);

// EXISTS subquery
@Query("SELECT d FROM Department d WHERE EXISTS " +
       "(SELECT e FROM Employee e WHERE e.department = d AND e.active = true)")
List<Department> findDepartmentsWithActiveEmployees();
```

## Complete Repository Example

```java
@Repository
public interface EmployeeRepository extends JpaRepository<Employee, Long> {

    // === Basic Queries ===
    @Query("SELECT e FROM Employee e WHERE e.active = true ORDER BY e.name")
    List<Employee> findAllActive();

    @Query("SELECT e FROM Employee e WHERE e.salary BETWEEN :min AND :max")
    List<Employee> findBySalaryRange(@Param("min") BigDecimal min,
                                     @Param("max") BigDecimal max);

    // === String Search ===
    @Query("SELECT e FROM Employee e " +
           "WHERE LOWER(e.name) LIKE LOWER(CONCAT('%', :keyword, '%'))")
    List<Employee> searchByName(@Param("keyword") String keyword);

    // === Join Queries ===
    @Query("SELECT e FROM Employee e JOIN FETCH e.department " +
           "WHERE e.active = true")
    List<Employee> findActiveWithDepartment();

    @Query("SELECT e FROM Employee e JOIN e.department d " +
           "WHERE d.name = :dept AND e.salary > :min")
    List<Employee> findByDepartmentAndMinSalary(@Param("dept") String dept,
                                                @Param("min") BigDecimal min);

    // === Aggregate Queries ===
    @Query("SELECT AVG(e.salary) FROM Employee e WHERE e.active = true")
    Double findAverageActiveSalary();

    @Query("SELECT d.name, COUNT(e), AVG(e.salary) " +
           "FROM Employee e JOIN e.department d " +
           "GROUP BY d.name ORDER BY COUNT(e) DESC")
    List<Object[]> getDepartmentStats();

    // === Constructor Expression (DTO) ===
    @Query("SELECT new com.example.hibernatedemo.dto.EmployeeSummary(" +
           "e.name, d.name, e.salary) " +
           "FROM Employee e JOIN e.department d " +
           "ORDER BY e.salary DESC")
    List<EmployeeSummary> findEmployeeSummaries();

    // === Subquery ===
    @Query("SELECT e FROM Employee e WHERE e.salary > " +
           "(SELECT AVG(e2.salary) FROM Employee e2)")
    List<Employee> findAboveAverageSalary();

    // === Bulk Operations ===
    @Modifying(clearAutomatically = true)
    @Query("UPDATE Employee e SET e.salary = e.salary * :factor " +
           "WHERE e.department.name = :dept")
    int adjustSalaries(@Param("factor") BigDecimal factor,
                       @Param("dept") String dept);

    @Modifying
    @Query("DELETE FROM Employee e WHERE e.active = false " +
           "AND e.hireDate < :cutoff")
    int purgeOldInactive(@Param("cutoff") LocalDate cutoff);
}
```

## Common Mistakes

1. **Using table/column names instead of entity/field names**: JPQL uses Java entity names and field names, not database names. `SELECT * FROM employees` is SQL. `SELECT e FROM Employee e` is JPQL.

2. **Forgetting the alias**: `SELECT FROM Employee WHERE active = true` fails. JPQL requires an alias: `SELECT e FROM Employee e WHERE e.active = true`.

3. **Using SELECT * in JPQL**: JPQL does not support `SELECT *`. Use `SELECT e` to select the full entity, or `SELECT e.name, e.email` for specific fields.

4. **Missing @Modifying on UPDATE/DELETE queries**: Without `@Modifying`, Spring Data expects a SELECT query and fails with an error.

5. **Forgetting @Param for named parameters**: Each `:paramName` in the query must have a matching `@Param("paramName")` on the method parameter.

6. **Using JOIN FETCH with pagination**: `JOIN FETCH` with `Pageable` causes Hibernate to apply pagination in memory (not in SQL), which loads all data first. Use `@EntityGraph` instead for paginated fetch joins.

7. **Not using constructor expressions for multi-field results**: Returning `List<Object[]>` is fragile and error-prone. Use constructor expressions (`new DTO(...)`) for type-safe results.

---

## Best Practices

1. **Use named parameters (:name) over positional (?1)**: They are more readable and safe against parameter reordering.

2. **Use constructor expressions for projections**: When you need only specific fields, return a DTO with `SELECT new DTO(...)` instead of `Object[]`.

3. **Use JOIN FETCH to prevent N+1**: When loading entities with relationships that you will access, use `JOIN FETCH` to load everything in one query.

4. **Use @Modifying for bulk operations**: Bulk UPDATE and DELETE are much faster than loading and modifying entities one by one.

5. **Keep queries in @Query annotations**: Placing JPQL directly on repository methods keeps the query close to its usage. Use `@NamedQuery` only when queries need to be shared.

6. **Test your queries**: Always check the generated SQL (via logging) to ensure it matches your intent, especially for complex joins and subqueries.

## Summary

In this chapter, you mastered JPQL — the standard query language for JPA:

- **JPQL queries entities and fields**, not tables and columns. It is database-independent and object-oriented.

- **@Query** on repository methods lets you write custom JPQL when derived methods are insufficient.

- **Named parameters** (`:name` with `@Param`) are preferred over positional parameters (`?1`) for readability and safety.

- **JOIN and JOIN FETCH** navigate entity relationships. JOIN FETCH loads related entities in a single query, preventing the N+1 problem.

- **Aggregate functions** (COUNT, SUM, AVG, MIN, MAX) with GROUP BY and HAVING enable statistical queries.

- **Constructor expressions** (`SELECT new DTO(...)`) return type-safe DTOs instead of fragile `Object[]` arrays.

- **@NamedQuery** defines reusable queries on the entity class, validated at startup.

- **Bulk UPDATE and DELETE** with `@Modifying` execute single SQL statements without loading entities.

- **JPQL functions** (CONCAT, UPPER, LOWER, CASE, COALESCE) handle string manipulation, math, dates, and conditional logic.

## Interview Questions

**Q1: What is the difference between JPQL and SQL?**

JPQL queries entity classes and their fields using Java names (e.g., `SELECT e FROM Employee e`). SQL queries database tables and columns (e.g., `SELECT * FROM employees`). JPQL is database-independent — Hibernate translates it to the appropriate SQL dialect. JPQL does not support `SELECT *`, requires aliases, and returns entity objects rather than raw rows.

**Q2: What is a constructor expression in JPQL and why use it?**

A constructor expression uses `SELECT new com.example.DTO(field1, field2)` to create DTO instances directly from query results. It provides type-safe results with proper field names instead of fragile `Object[]` arrays with index-based access. The DTO must have a constructor matching the selected fields exactly.

**Q3: What is the difference between JOIN and JOIN FETCH in JPQL?**

`JOIN` creates a SQL join and can filter/select based on the joined entity, but does not eagerly load the relationship. `JOIN FETCH` creates a SQL join AND loads the related entity data into the persistence context, preventing lazy loading and the N+1 problem. Use `JOIN` when you only need to filter; use `JOIN FETCH` when you will access the related entity's data.

**Q4: How do named parameters differ from positional parameters?**

Named parameters use `:paramName` syntax and require `@Param("paramName")` on method parameters. Positional parameters use `?1`, `?2` and match by method parameter order. Named parameters are more readable, self-documenting, and safe against parameter reordering. They are the recommended approach.

**Q5: What does @Modifying do and what are its limitations?**

`@Modifying` marks a `@Query` method as an UPDATE or DELETE operation. It executes a single SQL statement and returns the number of affected rows. Limitations: it does not trigger JPA lifecycle callbacks (@PreUpdate, @PreRemove), it does not update the persistence context (use `clearAutomatically = true`), and it requires a `@Transactional` context.

**Q6: Can JPQL query across multiple entity relationships?**

Yes. JPQL supports navigating relationships with dot notation (e.g., `e.department.name`) and explicit joins (`JOIN e.department d`). You can chain multiple joins: `SELECT e FROM Employee e JOIN e.department d JOIN d.company c WHERE c.name = :company`. Hibernate generates the appropriate SQL JOINs.

**Q7: What is the difference between @Query and @NamedQuery?**

`@Query` is placed on a repository method and associates the JPQL with that specific method. `@NamedQuery` is placed on the entity class and defines a named query that can be referenced by convention (`EntityName.methodName`). Both are validated at startup. `@Query` is more common because it keeps the query close to its usage.

**Q8: How do aggregate functions with GROUP BY work in JPQL?**

Aggregate functions (COUNT, SUM, AVG, MIN, MAX) operate on groups defined by GROUP BY. `SELECT d.name, COUNT(e) FROM Employee e JOIN e.department d GROUP BY d.name` counts employees per department. HAVING filters groups after aggregation: `HAVING COUNT(e) > 5` keeps only departments with more than 5 employees. Results are typically returned as `Object[]` or DTOs via constructor expressions.
