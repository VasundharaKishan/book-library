## Learning Goals

By the end of this chapter, you will be able to:

- Use @Column to control column names, length, precision, nullability, and more
- Map Java enums to database columns with @Enumerated
- Handle date and time fields with Java 8+ types
- Store large text and binary data with @Lob
- Exclude fields from persistence with @Transient
- Use @Basic to control fetch behavior on individual fields
- Understand implicit vs physical naming strategies
- Use Hibernate-specific annotations: @NaturalId, @Formula, @DynamicUpdate, @DynamicInsert
- Know which annotations are JPA standard and which are Hibernate-specific

## @Entity and @Table

You have already seen these in Chapter 3. Let us understand their full options.

### @Entity

```java
@Entity                          // Uses class name as entity name
@Entity(name = "ProductEntity")  // Custom entity name (used in JPQL)
```

The `name` attribute of `@Entity` sets the entity name used in **JPQL queries**, not the table name:

```java
@Entity(name = "ProductEntity")
public class Product { ... }

// In JPQL, you use the entity name:
// SELECT p FROM ProductEntity p WHERE p.price > 100
```

Most of the time, you do not need to set the entity name. The default (class name) is fine.

**JPA Standard**: `@Entity` is defined in `jakarta.persistence`.

### @Table

```java
@Table(
    name = "products",               // Table name
    schema = "inventory",            // Database schema (optional)
    catalog = "mydb",                // Database catalog (optional)
    uniqueConstraints = {            // Table-level unique constraints
        @UniqueConstraint(
            name = "uk_product_code",
            columnNames = {"code"}
        ),
        @UniqueConstraint(
            name = "uk_name_category",
            columnNames = {"name", "category"}  // Composite unique
        )
    },
    indexes = {                      // Table indexes
        @Index(
            name = "idx_product_name",
            columnList = "name"
        ),
        @Index(
            name = "idx_category_price",
            columnList = "category, price DESC"
        )
    }
)
```

**When to use @Table**: Always use it to explicitly set the table name. Without it, Hibernate uses a naming strategy to derive the table name from the class name, which can produce unexpected results (e.g., `OrderItem` might become `order_item` or `ORDERITEM` depending on the naming strategy).

**JPA Standard**: `@Table` is defined in `jakarta.persistence`.

## @Column — Controlling Column Mapping

`@Column` is the most frequently used annotation after `@Entity` and `@Id`. It controls how a Java field maps to a database column.

### All @Column Attributes

```java
@Column(
    name = "product_name",    // Column name in the database
    nullable = false,         // NOT NULL constraint (default: true)
    unique = false,           // UNIQUE constraint (default: false)
    length = 200,             // VARCHAR length (default: 255)
    precision = 10,           // Total digits for DECIMAL (default: 0)
    scale = 2,                // Decimal places for DECIMAL (default: 0)
    insertable = true,        // Include in INSERT statements (default: true)
    updatable = true,         // Include in UPDATE statements (default: true)
    columnDefinition = "",    // Raw SQL column definition (overrides everything)
    table = ""                // Secondary table name (advanced)
)
```

### Practical Examples

**String fields:**

```java
// Short string with constraints
@Column(name = "username", nullable = false, unique = true, length = 50)
private String username;
// SQL: username VARCHAR(50) NOT NULL UNIQUE

// Default mapping (no @Column needed)
private String description;
// SQL: description VARCHAR(255)

// Long string (but not @Lob)
@Column(length = 2000)
private String bio;
// SQL: bio VARCHAR(2000)
```

**Numeric fields:**

```java
// Money/decimal fields — always specify precision and scale
@Column(precision = 10, scale = 2)
private BigDecimal price;
// SQL: price DECIMAL(10,2)
// Stores values like 99999999.99

@Column(precision = 5, scale = 4)
private BigDecimal taxRate;
// SQL: tax_rate DECIMAL(5,4)
// Stores values like 0.0825

// Integer fields usually need no @Column
private int quantity;
// SQL: quantity INTEGER NOT NULL
// (primitives are NOT NULL by default)
```

**Read-only fields:**

```java
// A field set only during INSERT, never updated
@Column(updatable = false)
private LocalDateTime createdAt;
// SQL generates: INSERT includes createdAt, UPDATE excludes it

// A field that comes from a database trigger or default
@Column(insertable = false, updatable = false)
private LocalDateTime lastModified;
// Never included in INSERT or UPDATE — managed by the database
```

**Raw column definition:**

```java
// Override everything with raw SQL
@Column(columnDefinition = "TEXT DEFAULT 'No description'")
private String notes;
// SQL: notes TEXT DEFAULT 'No description'

// Database-specific type
@Column(columnDefinition = "JSONB")
private String metadata;
// SQL: metadata JSONB (PostgreSQL specific)
```

### When @Column Is Optional

You do not need `@Column` if you are happy with the defaults:

```
Field Type           Default Column Type    Default Nullable    Default Length
-----------------------------------------------------------------------------
String               VARCHAR(255)           true                255
int (primitive)      INTEGER                false (NOT NULL)    N/A
Integer (wrapper)    INTEGER                true                N/A
long (primitive)     BIGINT                 false (NOT NULL)    N/A
Long (wrapper)       BIGINT                 true                N/A
boolean              BOOLEAN                false               N/A
Boolean              BOOLEAN                true                N/A
double               DOUBLE                 false               N/A
BigDecimal           DECIMAL(38,2)*         true                N/A
LocalDate            DATE                   true                N/A
LocalDateTime        TIMESTAMP              true                N/A
```

*BigDecimal default precision varies by database dialect.

**Key rule**: Primitive types (`int`, `long`, `boolean`, `double`) are always NOT NULL. Wrapper types (`Integer`, `Long`, `Boolean`, `Double`) are nullable by default. Use wrapper types when null is a valid value.

```java
// primitive: NOT NULL in database (use when null is never valid)
private int age;

// wrapper: nullable in database (use when null is meaningful)
private Integer age;  // null could mean "age not provided"
```

**JPA Standard**: `@Column` is defined in `jakarta.persistence`.

## @Enumerated — Mapping Java Enums

Java enums are a natural way to represent fixed sets of values. JPA provides `@Enumerated` to map them to database columns.

### Two Strategies: ORDINAL vs STRING

```java
public enum Status {
    DRAFT,      // ordinal 0
    ACTIVE,     // ordinal 1
    ARCHIVED    // ordinal 2
}
```

```java
// ORDINAL (default) — stores the enum's position number
@Enumerated(EnumType.ORDINAL)
private Status status;
// Database value: 0, 1, or 2
// SQL: status INTEGER

// STRING — stores the enum's name as text
@Enumerated(EnumType.STRING)
private Status status;
// Database value: "DRAFT", "ACTIVE", or "ARCHIVED"
// SQL: status VARCHAR(255)
```

### Why You Should Always Use STRING

```
+---------------------------------------------------------------+
|   ORDINAL                        STRING                        |
|   (AVOID)                        (RECOMMENDED)                 |
|                                                                |
|   Stores: 0, 1, 2               Stores: "DRAFT", "ACTIVE",    |
|                                          "ARCHIVED"            |
|                                                                |
|   Problem: If you add a new     No problem: Adding new values  |
|   enum value in the middle:     anywhere is safe:              |
|                                                                |
|   enum Status {                 enum Status {                  |
|     DRAFT,     // 0             DRAFT,                         |
|     REVIEW,    // 1 (NEW!)      REVIEW,    (NEW!)              |
|     ACTIVE,    // 2 (was 1!)    ACTIVE,                        |
|     ARCHIVED   // 3 (was 2!)    ARCHIVED                       |
|   }                             }                              |
|                                                                |
|   All existing "1" values now   "ACTIVE" still means ACTIVE    |
|   mean REVIEW instead of        regardless of position.        |
|   ACTIVE! DATA CORRUPTION!                                     |
+---------------------------------------------------------------+
```

**Always use `EnumType.STRING`**. It is slightly less storage-efficient (strings vs integers), but it is safe against reordering and self-documenting when you query the database directly.

### Complete Enum Example

```java
public enum Priority {
    LOW, MEDIUM, HIGH, CRITICAL
}

public enum Category {
    ELECTRONICS, CLOTHING, BOOKS, FOOD, HOME
}

@Entity
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private Category category;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 10)
    private Priority priority;

    // constructors, getters, setters...
}
```

Generated SQL:
```sql
CREATE TABLE products (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    name VARCHAR(255) NOT NULL,
    category VARCHAR(20) NOT NULL,
    priority VARCHAR(10) NOT NULL,
    PRIMARY KEY (id)
);

-- Sample data:
INSERT INTO products (name, category, priority)
VALUES ('Laptop', 'ELECTRONICS', 'HIGH');
```

**JPA Standard**: `@Enumerated` is defined in `jakarta.persistence`.

## Date and Time Mapping

### Java 8+ Date/Time Types (Recommended)

With JPA 2.2+ and Hibernate 6, Java 8 date/time types are supported directly — no extra annotations needed:

```java
@Entity
public class Event {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    // DATE only (no time) — e.g., 2024-03-15
    private LocalDate eventDate;
    // SQL: event_date DATE

    // TIME only (no date) — e.g., 14:30:00
    private LocalTime startTime;
    // SQL: start_time TIME

    // DATE + TIME (no timezone) — e.g., 2024-03-15T14:30:00
    private LocalDateTime createdAt;
    // SQL: created_at TIMESTAMP

    // DATE + TIME + TIMEZONE OFFSET — e.g., 2024-03-15T14:30:00+05:30
    private OffsetDateTime publishedAt;
    // SQL: published_at TIMESTAMP WITH TIME ZONE

    // INSTANT — a point on the timeline (UTC)
    private Instant lastModified;
    // SQL: last_modified TIMESTAMP WITH TIME ZONE
}
```

### Java Type to SQL Type Mapping

```
Java Type                  SQL Type                    Example Value
----------------------------------------------------------------------
LocalDate                  DATE                        2024-03-15
LocalTime                  TIME                        14:30:00
LocalDateTime              TIMESTAMP                   2024-03-15 14:30:00
OffsetDateTime             TIMESTAMP WITH TIME ZONE    2024-03-15 14:30:00+05:30
Instant                    TIMESTAMP WITH TIME ZONE    2024-03-15 09:00:00Z
ZonedDateTime              TIMESTAMP WITH TIME ZONE    2024-03-15 14:30:00+05:30
Year                       INTEGER                     2024
YearMonth                  Not directly supported       (use converter)
```

### Legacy Date Types and @Temporal

If you work with older code, you might encounter `java.util.Date` or `java.util.Calendar`. These require `@Temporal`:

```java
// LEGACY — avoid in new code
@Temporal(TemporalType.DATE)        // Store date only
private java.util.Date birthDate;

@Temporal(TemporalType.TIME)        // Store time only
private java.util.Date startTime;

@Temporal(TemporalType.TIMESTAMP)   // Store date + time
private java.util.Date createdAt;

// MODERN — prefer this in new code
private LocalDate birthDate;         // No @Temporal needed
private LocalTime startTime;         // No @Temporal needed
private LocalDateTime createdAt;     // No @Temporal needed
```

**Best practice**: Always use `java.time.*` types (LocalDate, LocalDateTime, Instant). They are clearer, immutable, and do not require `@Temporal`.

**JPA Standard**: `@Temporal` is defined in `jakarta.persistence`. Java 8+ date/time support is part of JPA 2.2+.

## @Lob — Large Objects

`@Lob` marks a field as a Large Object, which maps to database types designed for large content.

```java
@Entity
public class Article {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    // Large text content
    @Lob
    private String content;
    // SQL: content CLOB (Character Large Object)
    // Can store megabytes of text

    // Binary data (images, files)
    @Lob
    private byte[] thumbnail;
    // SQL: thumbnail BLOB (Binary Large Object)
    // Can store binary data
}
```

### CLOB vs BLOB

```
+---------------------------------------------------------------+
|  @Lob on String          |  @Lob on byte[]                    |
|  (CLOB)                  |  (BLOB)                            |
|                          |                                     |
|  Character Large Object  |  Binary Large Object                |
|  Stores: text content    |  Stores: images, PDFs, files       |
|  Size: up to 4 GB        |  Size: up to 4 GB                  |
|  SQL: CLOB / TEXT         |  SQL: BLOB / BYTEA                 |
|                          |                                     |
|  Use for:                |  Use for:                           |
|  - Article bodies        |  - Profile pictures                 |
|  - HTML content          |  - File attachments                 |
|  - JSON/XML documents    |  - Thumbnails                       |
+---------------------------------------------------------------+
```

**Caution**: Loading LOB data is expensive. If your entity has a large LOB field that you do not always need, consider lazy loading (with `@Basic(fetch = FetchType.LAZY)`) or moving the LOB to a separate entity.

**JPA Standard**: `@Lob` is defined in `jakarta.persistence`.

## @Transient — Excluding Fields

`@Transient` tells JPA to ignore a field — it will not be mapped to any database column.

```java
@Entity
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @Column(precision = 10, scale = 2)
    private BigDecimal price;

    @Column(precision = 5, scale = 4)
    private BigDecimal taxRate;

    // NOT stored in database — calculated at runtime
    @Transient
    private BigDecimal priceWithTax;

    // NOT stored in database — used only in application logic
    @Transient
    private boolean selected;

    // Calculate the transient field
    public BigDecimal getPriceWithTax() {
        if (price != null && taxRate != null) {
            return price.multiply(BigDecimal.ONE.add(taxRate));
        }
        return price;
    }
}
```

### Three Ways to Exclude Fields

```
Method                   Annotation Required?    Works With
--------------------------------------------------------------
@Transient               Yes                    JPA (standard)
transient keyword        No                     Java serialization
                                                AND JPA
static fields            No                     Never persisted
```

```java
@Transient                           // JPA ignores this field
private double discount;

private transient double discount;   // Both Java serialization AND JPA
                                     // ignore this field

private static int instanceCount;    // Static fields are never persisted
```

**Best practice**: Use the `@Transient` annotation (not the `transient` keyword) because it is explicit and only affects JPA, not Java serialization.

**JPA Standard**: `@Transient` is defined in `jakarta.persistence`.

## @Basic — Fetch Type and Optionality

`@Basic` controls two things: whether a field is loaded lazily and whether it is optional (nullable).

```java
// Default behavior (you rarely need to write this)
@Basic(fetch = FetchType.EAGER, optional = true)
private String name;

// Lazy loading for large fields
@Basic(fetch = FetchType.LAZY)
@Lob
private String content;
// Content is NOT loaded when the entity is fetched.
// It is loaded only when getContent() is called.

// Field is required (NOT NULL)
@Basic(optional = false)
private String email;
// Hibernate validates non-null before sending to database
```

### Lazy Loading Individual Fields

```
Default behavior (EAGER):
+--------------------------------------------------------------+
| SELECT id, name, email, content FROM articles WHERE id = 1   |
|                                                               |
| All columns loaded in one query, even if you only need       |
| the name. Large LOB columns waste bandwidth.                  |
+--------------------------------------------------------------+

With @Basic(fetch = LAZY) on content:
+--------------------------------------------------------------+
| Query 1: SELECT id, name, email FROM articles WHERE id = 1   |
|          (content NOT loaded)                                 |
|                                                               |
| Query 2 (only if getContent() is called):                    |
|          SELECT content FROM articles WHERE id = 1            |
+--------------------------------------------------------------+
```

**Important**: Lazy loading of basic fields requires Hibernate bytecode enhancement, which is not enabled by default in Spring Boot. Without it, `@Basic(fetch = LAZY)` is silently ignored and the field is loaded eagerly. To enable it, add to `pom.xml`:

```xml
<plugin>
    <groupId>org.hibernate.orm.tooling</groupId>
    <artifactId>hibernate-enhance-maven-plugin</artifactId>
    <version>${hibernate.version}</version>
    <executions>
        <execution>
            <configuration>
                <enableLazyInitialization>true</enableLazyInitialization>
            </configuration>
            <goals>
                <goal>enhance</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

For most applications, a simpler approach is to move large fields to a separate entity with a one-to-one relationship (covered in Chapter 12).

**JPA Standard**: `@Basic` is defined in `jakarta.persistence`.

## Naming Strategies

When you do not explicitly specify a column or table name, Hibernate uses a **naming strategy** to derive the name from the Java field or class name.

### How Names Are Derived

```
Java Name                  Implicit Strategy          Physical Strategy
                           (logical name)             (database name)
----------------------------------------------------------------------
Class: OrderItem           OrderItem                  order_item
Field: firstName           firstName                  first_name
Field: createdAt           createdAt                  created_at
Field: isActive            isActive                   is_active
Class: HTMLParser          HTMLParser                  htmlparser
```

Spring Boot uses these default strategies:

```
+----------------------------------------------------------------+
|  Naming Strategy Flow                                           |
|                                                                 |
|  Java name: "orderDate"                                         |
|       |                                                         |
|       v                                                         |
|  Implicit Strategy (SpringImplicitNamingStrategy):              |
|       Derives logical name from Java name                       |
|       "orderDate" --> "orderDate"                               |
|       |                                                         |
|       v                                                         |
|  Physical Strategy (CamelCaseToUnderscoresNamingStrategy):      |
|       Converts logical name to database name                    |
|       "orderDate" --> "order_date"                               |
|       |                                                         |
|       v                                                         |
|  Database column: order_date                                    |
+----------------------------------------------------------------+
```

### Overriding with @Column and @Table

When you specify a name explicitly, the naming strategy is bypassed:

```java
@Column(name = "order_date")     // Exact name — no naming strategy applied
private LocalDate orderDate;

@Table(name = "order_items")     // Exact name — no naming strategy applied
public class OrderItem { ... }
```

### Changing the Naming Strategy

You can change the naming strategy in `application.properties`:

```properties
# Use exact Java names (no underscore conversion)
spring.jpa.hibernate.naming.physical-strategy=org.hibernate.boot.model.naming.PhysicalNamingStrategyStandardImpl

# With this strategy:
# orderDate --> orderDate (not order_date)
# OrderItem --> OrderItem (not order_item)
```

**Best practice**: Stick with the Spring Boot default (camelCase to snake_case) and explicitly name columns with `@Column(name = "...")` when the default does not match your database conventions.

## Hibernate-Specific Annotations

The following annotations are **not part of the JPA specification**. They are provided by Hibernate and will not work with other JPA providers (EclipseLink, OpenJPA). Use them when JPA does not offer what you need, but be aware of the trade-off.

### @NaturalId — Business Keys

A **natural ID** (or business key) is a unique identifier that has real-world meaning, as opposed to the surrogate primary key (the auto-generated `id`).

```java
import org.hibernate.annotations.NaturalId;

@Entity
@Table(name = "books")
public class Book {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;             // Surrogate key (database-generated)

    @NaturalId                   // Hibernate: Business key
    @Column(nullable = false, unique = true, length = 13)
    private String isbn;         // Natural key (real-world identifier)

    @Column(nullable = false)
    private String title;

    // ...
}
```

```
+---------------------------------------------------------------+
|  Surrogate Key (id)           vs.     Natural Key (isbn)       |
|                                                                |
|  - Auto-generated by DB               - Has real-world meaning |
|  - No business meaning                - ISBN, email, SSN, etc. |
|  - Always unique                      - Always unique          |
|  - Used for JOINs and FKs             - Used for lookups       |
|  - Never changes                      - Rarely changes         |
+---------------------------------------------------------------+
```

With `@NaturalId`, Hibernate provides optimized lookup methods:

```java
// Standard JPA way (by surrogate key)
Book book = entityManager.find(Book.class, 1L);

// Hibernate way (by natural key) — uses Session, not EntityManager
Session session = entityManager.unwrap(Session.class);
Book book = session.byNaturalId(Book.class)
    .using("isbn", "978-0-13-468599-1")
    .load();
```

Hibernate also caches natural ID lookups in the first-level cache, so repeated lookups by the same natural ID do not hit the database.

**Hibernate-specific**: `@NaturalId` is from `org.hibernate.annotations`.

### @Formula — Computed Columns

`@Formula` lets you define a field whose value is computed by a SQL expression rather than stored in a column:

```java
import org.hibernate.annotations.Formula;

@Entity
@Table(name = "order_items")
public class OrderItem {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private int quantity;

    @Column(precision = 10, scale = 2)
    private BigDecimal unitPrice;

    // Computed field — NOT a real column
    @Formula("quantity * unit_price")
    private BigDecimal totalPrice;

    // Another example: computed from a subquery
    @Formula("(SELECT COUNT(*) FROM reviews r WHERE r.product_id = id)")
    private int reviewCount;
}
```

```
Database Table: order_items
+----+----------+------------+
| id | quantity | unit_price |
|----|----------|------------|
| 1  | 3        | 29.99      |
| 2  | 1        | 149.99     |
+----+----------+------------+
Note: total_price is NOT a column — it does not exist in the table.

When Hibernate loads an OrderItem, it generates:
SELECT id, quantity, unit_price,
       quantity * unit_price AS formula_0   <-- @Formula
FROM order_items WHERE id = 1

Result: totalPrice = 89.97 (3 * 29.99)
```

**Key points about @Formula:**
- The expression uses **SQL column names** (not Java field names)
- The formula is read-only — you cannot set or update it
- The formula is evaluated by the database, so it supports SQL functions
- The value is calculated every time the entity is loaded (no caching)

**Hibernate-specific**: `@Formula` is from `org.hibernate.annotations`.

### @DynamicUpdate — Efficient Updates

By default, Hibernate generates UPDATE statements that set **all** columns, even those that have not changed:

```sql
-- Default behavior: UPDATE ALL columns
UPDATE products
SET name = 'Widget', price = 29.99, description = 'A widget',
    category = 'TOOLS', status = 'ACTIVE', created_at = '2024-01-01'
WHERE id = 1
-- Even if only the price changed!
```

`@DynamicUpdate` tells Hibernate to generate UPDATE statements that only include changed columns:

```java
import org.hibernate.annotations.DynamicUpdate;

@Entity
@DynamicUpdate          // Only update changed columns
@Table(name = "products")
public class Product {
    // ...
}
```

```sql
-- With @DynamicUpdate: Only changed columns
UPDATE products
SET price = 29.99
WHERE id = 1
-- Only the price column!
```

### When to Use @DynamicUpdate

```
+----------------------------------------------------------------+
|  Use @DynamicUpdate when:          | Skip when:                  |
|                                    |                             |
|  - Table has many columns          | - Table has few columns     |
|    (15+) and updates are           |   (updating all is fine)    |
|    typically partial               |                             |
|  - Table has indexed columns       | - Every update changes      |
|    that should not be              |   most columns              |
|    unnecessarily touched           |                             |
|  - You need to avoid trigger       | - Simplicity is more        |
|    firing on unchanged columns     |   important than the minor  |
|                                    |   performance gain          |
+----------------------------------------------------------------+
```

**Hibernate-specific**: `@DynamicUpdate` is from `org.hibernate.annotations`.

### @DynamicInsert — Efficient Inserts

Similar to `@DynamicUpdate`, `@DynamicInsert` generates INSERT statements that skip null fields (letting the database use column defaults):

```java
import org.hibernate.annotations.DynamicInsert;

@Entity
@DynamicInsert
@Table(name = "products")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    // Database has: status VARCHAR(20) DEFAULT 'DRAFT'
    @Column(length = 20)
    private String status;

    // ...
}
```

```sql
-- Without @DynamicInsert (default)
INSERT INTO products (name, status) VALUES ('Widget', NULL)
-- status becomes NULL, ignoring the database DEFAULT

-- With @DynamicInsert
INSERT INTO products (name) VALUES ('Widget')
-- status is omitted, so the database applies DEFAULT 'DRAFT'
```

**Hibernate-specific**: `@DynamicInsert` is from `org.hibernate.annotations`.

## Complete Entity Example

Let us put everything together in a realistic entity:

```java
package com.example.hibernatedemo.entity;

import jakarta.persistence.*;
import org.hibernate.annotations.DynamicUpdate;
import org.hibernate.annotations.Formula;
import org.hibernate.annotations.NaturalId;

import java.math.BigDecimal;
import java.time.LocalDate;
import java.time.LocalDateTime;

@Entity
@Table(
    name = "products",
    uniqueConstraints = {
        @UniqueConstraint(name = "uk_product_sku", columnNames = "sku")
    },
    indexes = {
        @Index(name = "idx_product_category", columnList = "category"),
        @Index(name = "idx_product_name", columnList = "name")
    }
)
@DynamicUpdate                                          // Hibernate: efficient updates
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NaturalId                                          // Hibernate: business key
    @Column(nullable = false, unique = true, length = 20)
    private String sku;                                 // e.g., "PROD-001"

    @Column(nullable = false, length = 200)
    private String name;

    @Lob                                                // JPA: large text
    private String description;

    @Column(nullable = false, precision = 10, scale = 2)
    private BigDecimal price;

    @Column(precision = 10, scale = 2)
    private BigDecimal costPrice;

    @Formula("price - cost_price")                      // Hibernate: computed field
    private BigDecimal profit;

    @Enumerated(EnumType.STRING)                        // JPA: enum as string
    @Column(nullable = false, length = 20)
    private Category category;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 15)
    private Status status;

    @Column(nullable = false)
    private int stockQuantity;

    @Column(nullable = false)
    private int reorderLevel;

    @Formula("CASE WHEN stock_quantity <= reorder_level " +  // Hibernate: computed
             "THEN true ELSE false END")
    private boolean needsReorder;

    private LocalDate launchDate;                       // JPA 2.2+: no @Temporal needed

    @Column(updatable = false)                          // Set once, never updated
    private LocalDateTime createdAt;

    private LocalDateTime updatedAt;

    @Transient                                          // JPA: not persisted
    private boolean featured;

    // Enums
    public enum Category {
        ELECTRONICS, CLOTHING, BOOKS, FOOD, HOME, SPORTS
    }

    public enum Status {
        DRAFT, ACTIVE, DISCONTINUED, OUT_OF_STOCK
    }

    // Default constructor (required by JPA)
    public Product() {
    }

    // Convenience constructor
    public Product(String sku, String name, BigDecimal price,
                   Category category) {
        this.sku = sku;
        this.name = name;
        this.price = price;
        this.category = category;
        this.status = Status.DRAFT;
        this.stockQuantity = 0;
        this.reorderLevel = 10;
        this.createdAt = LocalDateTime.now();
        this.updatedAt = LocalDateTime.now();
    }

    // Lifecycle callback — sets timestamps automatically
    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
        this.updatedAt = LocalDateTime.now();
    }

    @PreUpdate
    protected void onUpdate() {
        this.updatedAt = LocalDateTime.now();
    }

    // Getters and setters
    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getSku() { return sku; }
    public void setSku(String sku) { this.sku = sku; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getDescription() { return description; }
    public void setDescription(String description) { this.description = description; }
    public BigDecimal getPrice() { return price; }
    public void setPrice(BigDecimal price) { this.price = price; }
    public BigDecimal getCostPrice() { return costPrice; }
    public void setCostPrice(BigDecimal costPrice) { this.costPrice = costPrice; }
    public BigDecimal getProfit() { return profit; }
    public Category getCategory() { return category; }
    public void setCategory(Category category) { this.category = category; }
    public Status getStatus() { return status; }
    public void setStatus(Status status) { this.status = status; }
    public int getStockQuantity() { return stockQuantity; }
    public void setStockQuantity(int stockQuantity) { this.stockQuantity = stockQuantity; }
    public int getReorderLevel() { return reorderLevel; }
    public void setReorderLevel(int reorderLevel) { this.reorderLevel = reorderLevel; }
    public boolean isNeedsReorder() { return needsReorder; }
    public LocalDate getLaunchDate() { return launchDate; }
    public void setLaunchDate(LocalDate launchDate) { this.launchDate = launchDate; }
    public LocalDateTime getCreatedAt() { return createdAt; }
    public LocalDateTime getUpdatedAt() { return updatedAt; }
    public boolean isFeatured() { return featured; }
    public void setFeatured(boolean featured) { this.featured = featured; }

    @Override
    public String toString() {
        return "Product{sku='" + sku + "', name='" + name +
               "', price=" + price + ", category=" + category +
               ", status=" + status + ", stock=" + stockQuantity + "}";
    }
}
```

### Generated SQL for This Entity

```sql
CREATE TABLE products (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    sku VARCHAR(20) NOT NULL,
    name VARCHAR(200) NOT NULL,
    description CLOB,
    price DECIMAL(10,2) NOT NULL,
    cost_price DECIMAL(10,2),
    category VARCHAR(20) NOT NULL,
    status VARCHAR(15) NOT NULL,
    stock_quantity INTEGER NOT NULL,
    reorder_level INTEGER NOT NULL,
    launch_date DATE,
    created_at TIMESTAMP,
    updated_at TIMESTAMP,
    PRIMARY KEY (id),
    CONSTRAINT uk_product_sku UNIQUE (sku)
);

CREATE INDEX idx_product_category ON products (category);
CREATE INDEX idx_product_name ON products (name);
```

Notice:
- `profit` and `needsReorder` are NOT columns — they are `@Formula` computed fields
- `featured` is NOT a column — it is `@Transient`
- `description` is `CLOB` because of `@Lob`
- `category` and `status` are `VARCHAR` because of `@Enumerated(STRING)`
- `created_at` and `updated_at` are `TIMESTAMP` (no `@Temporal` needed for `LocalDateTime`)

## Lifecycle Callbacks

JPA defines callback annotations that let you hook into entity lifecycle events:

```java
@Entity
public class AuditableEntity {

    @PrePersist     // Called before INSERT
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
    }

    @PostPersist    // Called after INSERT
    protected void afterCreate() {
        System.out.println("Entity created with ID: " + id);
    }

    @PreUpdate      // Called before UPDATE
    protected void onUpdate() {
        this.updatedAt = LocalDateTime.now();
    }

    @PostUpdate     // Called after UPDATE
    protected void afterUpdate() {
        System.out.println("Entity updated: " + id);
    }

    @PreRemove      // Called before DELETE
    protected void onDelete() {
        System.out.println("About to delete: " + id);
    }

    @PostRemove     // Called after DELETE
    protected void afterDelete() {
        System.out.println("Deleted: " + id);
    }

    @PostLoad       // Called after entity is loaded from DB
    protected void onLoad() {
        // Initialize transient fields
        this.displayName = firstName + " " + lastName;
    }
}
```

### Lifecycle Callback Flow

```
+----------------------------------------------------------------+
|  Operation: repository.save(new Entity())                       |
|                                                                 |
|  1. @PrePersist fires (set createdAt, validate, etc.)           |
|  2. INSERT INTO ... (SQL executes)                              |
|  3. @PostPersist fires (log creation, send notification, etc.)  |
+----------------------------------------------------------------+

+----------------------------------------------------------------+
|  Operation: entity.setName("new"); repository.save(entity)      |
|                                                                 |
|  1. @PreUpdate fires (set updatedAt, validate, etc.)            |
|  2. UPDATE ... SET ... (SQL executes)                           |
|  3. @PostUpdate fires (log update, send notification, etc.)     |
+----------------------------------------------------------------+

+----------------------------------------------------------------+
|  Operation: repository.deleteById(1L)                           |
|                                                                 |
|  1. SELECT ... (load the entity)                                |
|  2. @PostLoad fires (entity is loaded)                          |
|  3. @PreRemove fires (cleanup, validation, etc.)                |
|  4. DELETE FROM ... (SQL executes)                              |
|  5. @PostRemove fires (log deletion, etc.)                      |
+----------------------------------------------------------------+
```

**Common use case**: Automatically setting `createdAt` and `updatedAt` timestamps without requiring the caller to remember:

```java
@PrePersist
protected void onCreate() {
    this.createdAt = LocalDateTime.now();
    this.updatedAt = LocalDateTime.now();
}

@PreUpdate
protected void onUpdate() {
    this.updatedAt = LocalDateTime.now();
}
```

We will explore a more sophisticated approach using Spring Data JPA's `@CreatedDate` and `@LastModifiedDate` in Chapter 21 (Auditing).

**JPA Standard**: All lifecycle callbacks (`@PrePersist`, `@PostPersist`, `@PreUpdate`, `@PostUpdate`, `@PreRemove`, `@PostRemove`, `@PostLoad`) are defined in `jakarta.persistence`.

## Common Mistakes

1. **Using `EnumType.ORDINAL` for enums**: Adding or reordering enum constants silently corrupts existing data. Always use `EnumType.STRING`.

2. **Using `java.util.Date` instead of `java.time.*`**: In new code, always use `LocalDate`, `LocalDateTime`, or `Instant`. They are immutable, type-safe, and do not require `@Temporal`.

3. **Forgetting that `@Formula` uses SQL column names**: The formula expression runs in SQL, so you must use database column names (e.g., `unit_price`), not Java field names (e.g., `unitPrice`).

4. **Expecting `@Lob` fields to lazy-load automatically**: Without Hibernate bytecode enhancement, `@Lob` fields are loaded eagerly. This can cause performance issues when loading entities with large text/binary fields.

5. **Not specifying `length` for String columns**: Without `@Column(length = ...)`, all Strings default to VARCHAR(255). For names, use 100. For emails, use 150. For descriptions, use 2000 or `@Lob`. Being intentional about sizes catches bugs early.

6. **Using primitive types when null is valid**: `private int age` is NOT NULL in the database. If age can be unknown/optional, use `private Integer age` instead.

7. **Confusing `@Entity(name)` with `@Table(name)`**: `@Entity(name)` sets the entity name for JPQL queries. `@Table(name)` sets the database table name. They serve different purposes.

## Best Practices

1. **Always use `@Enumerated(EnumType.STRING)`**: The minor storage cost is worth the safety against enum reordering.

2. **Be explicit about column constraints**: Even though JPA infers some things, always specify `nullable`, `length`, and `unique` when they matter. It makes your code self-documenting and catches mistakes at DDL time rather than runtime.

3. **Use `@Column(updatable = false)` for creation timestamps**: This prevents accidental modification of fields that should be set once.

4. **Prefer JPA standard annotations**: Use `jakarta.persistence.*` annotations whenever possible. Only use `org.hibernate.annotations.*` when JPA does not provide what you need.

5. **Use lifecycle callbacks for automatic timestamps**: `@PrePersist` and `@PreUpdate` ensure timestamps are always set, even if the calling code forgets.

6. **Document Hibernate-specific annotations**: When you use `@NaturalId`, `@Formula`, or `@DynamicUpdate`, add a comment noting it is Hibernate-specific. This helps if you ever need to switch providers.

## Summary

In this chapter, you learned every major annotation for mapping Java classes to database tables:

- **@Column** controls column name, length, precision, scale, nullability, uniqueness, and whether the field is included in INSERT/UPDATE statements.

- **@Enumerated(EnumType.STRING)** maps Java enums as readable strings. Always use STRING, never ORDINAL.

- **Java 8+ date/time types** (LocalDate, LocalDateTime, Instant) map directly to SQL DATE/TIMESTAMP without needing `@Temporal`.

- **@Lob** maps large text (CLOB) and binary (BLOB) data. Consider lazy loading for large fields.

- **@Transient** excludes a field from database persistence.

- **@Basic** controls fetch type (EAGER/LAZY) and optionality for individual fields.

- **Naming strategies** determine how Java names translate to database names. Spring Boot defaults to camelCase-to-snake_case.

- **@NaturalId** (Hibernate) marks a business key for optimized lookups.

- **@Formula** (Hibernate) computes a field value from a SQL expression.

- **@DynamicUpdate** (Hibernate) generates UPDATE statements with only changed columns.

- **Lifecycle callbacks** (@PrePersist, @PreUpdate, etc.) let you hook into entity operations for automatic timestamps and validation.

## Interview Questions

**Q1: What is the difference between `@Column(unique = true)` and `@Table(uniqueConstraints = ...)`?**

`@Column(unique = true)` creates a unique constraint on a single column. `@Table(uniqueConstraints = @UniqueConstraint(columnNames = {"col1", "col2"}))` can create composite unique constraints spanning multiple columns. Use `@Column` for single-column uniqueness and `@Table` for multi-column uniqueness or when you need to name the constraint.

**Q2: Why should you always use `EnumType.STRING` instead of `EnumType.ORDINAL`?**

`ORDINAL` stores the enum's position (0, 1, 2). If you add a new enum value or reorder them, existing data becomes corrupted because the numbers now map to different values. `STRING` stores the name ("ACTIVE", "DRAFT"), which is independent of position and remains correct even after reordering.

**Q3: What is the difference between `@Transient` and `@Formula`?**

`@Transient` marks a field that is completely ignored by JPA — it has no column and no SQL. `@Formula` marks a field that has no column but IS populated by a SQL expression when the entity is loaded. `@Formula` fields are read-only and computed by the database.

**Q4: What is `@NaturalId` and when would you use it?**

`@NaturalId` is a Hibernate-specific annotation that marks a field as a business key (like ISBN, email, or SKU). Unlike the primary key (`@Id`), a natural ID has real-world meaning. Hibernate provides optimized lookup methods for natural IDs and caches them in the first-level cache. Use it when you frequently look up entities by a business identifier.

**Q5: How do primitive vs wrapper types affect nullability in JPA?**

Primitive types (`int`, `long`, `boolean`) cannot be null in Java, so JPA maps them to NOT NULL columns. Wrapper types (`Integer`, `Long`, `Boolean`) can be null, so JPA maps them to nullable columns by default. Use primitives when null is never a valid value, and wrappers when null represents "unknown" or "not set."

**Q6: What is the purpose of `@Column(insertable = false, updatable = false)`?**

A field with both `insertable = false` and `updatable = false` is never included in INSERT or UPDATE SQL statements. The value comes from the database itself — perhaps a default value, a trigger, or a computed column. JPA reads the value but never writes it.

**Q7: What does `@DynamicUpdate` do and when should you use it?**

By default, Hibernate generates UPDATE statements that set all columns, even unchanged ones. `@DynamicUpdate` tells Hibernate to generate UPDATE statements that only include columns whose values have actually changed. Use it on entities with many columns (15+) where updates typically change only a few fields. This can improve performance and avoid unnecessary trigger firing on unchanged columns.

**Q8: What are JPA lifecycle callbacks? Name four of them.**

Lifecycle callbacks are methods annotated to run at specific points in an entity's lifecycle. `@PrePersist` runs before INSERT, `@PostPersist` runs after INSERT, `@PreUpdate` runs before UPDATE, `@PostUpdate` runs after UPDATE, `@PreRemove` runs before DELETE, `@PostRemove` runs after DELETE, and `@PostLoad` runs after an entity is loaded from the database. They are commonly used for setting timestamps and logging.
